# AI 讲题系统 MVP 产品规格

## 1. 产品目标

Phase 1 的目标是完成 AI 讲题系统的最小可用版本，让小学生能够通过“拍照或上传题目图片 + AI 启发式讲解”的方式完成基础学习辅导。

核心学习闭环：

```text
拍照/上传题目图片 -> OCR 识别 -> AI 讲解 -> 学生继续追问 -> AI 连续辅导 -> 生成学习记录
```

系统的重点不是直接给出答案，而是帮助孩子理解题目。AI 的表达需要像一位耐心的小学老师：语言简单、步骤清楚、一次只讲一小步，并通过鼓励式互动引导孩子继续思考。

## 2. 目标用户

- 需要完成作业辅导的小学生。
- 首期支持学科：数学、语文、英语。
- 优先支持场景：小学数学应用题、计算题、填空题等典型题型。

## 3. MVP 范围

### 3.1 本阶段包含

- 手机拍照上传。
- 手机相册上传。
- 桌面端拖拽上传。
- 支持 PNG、JPG 图片格式。
- 上传前进行基础图片压缩。
- 基础图片预处理，提高 OCR 识别效率。
- OCR 识别题目文字、数字、常见符号和简单公式。
- OCR 结果转换为标准 JSON 结构。
- AI 使用启发式讲题，而不是直接输出最终答案。
- 支持同一道题的多轮追问。
- 保存基础学习记录。
- 限制所有回答在小学教育场景内。

### 3.2 本阶段不包含

- 原生 iOS 或 Android App。
- 完整学生账号系统和家长端系统。
- 完整错题本系统。
- 高级学情分析。
- 长期个性化学习规划。
- 自研 OCR 模型训练。
- 复杂手写公式识别。
- 支付、会员、商业化能力。

## 4. 核心用户流程

1. 学生在手机或电脑上打开 Web 应用。
2. 学生拍照、从相册选择图片，或在电脑上拖拽题目图片。
3. 系统校验图片格式，并在上传前压缩图片。
4. 系统将图片上传到后端。
5. 后端对图片做基础预处理，并调用 OCR 服务。
6. OCR 结果被转换成统一的题目 JSON 结构。
7. 学生可以查看并手动修正识别出的题目文字。
8. 学生点击开始讲解，或输入“这题我不会”。
9. AI 按照启发式教学方式讲解题目。
10. 学生可以继续追问，例如“我还是不会”“为什么这样算”。
11. 系统保存题目、对话和学习记录。

## 5. 功能需求

### 5.1 图片上传

图片上传系统需要支持：

- 手机摄像头拍照上传。
- 手机和电脑本地文件选择。
- 桌面端拖拽上传。
- PNG 和 JPG 图片格式。
- 文件大小校验。
- 上传前客户端图片压缩。
- 上传前图片预览。
- 对不支持的格式或上传失败给出清晰提示。

建议接口：

```http
POST /api/question-images
```

建议返回：

```json
{
  "imageId": "img_123",
  "status": "uploaded",
  "width": 1280,
  "height": 960,
  "mimeType": "image/jpeg"
}
```

验收标准：

- 学生可以通过手机拍照上传题目。
- 学生可以从相册选择图片。
- 桌面端用户可以拖拽上传图片。
- PNG 和 JPG 文件可以正常上传。
- 非支持格式会被拒绝，并显示可理解的提示。
- 大图会在 OCR 前被压缩。

### 5.2 图片预处理

后端需要通过基础预处理提升 OCR 成功率：

- 保存原始图片。
- 生成用于 OCR 的处理后图片。
- 根据图片元信息做基础方向纠正。
- 做基础尺寸调整或压缩。
- 在有必要时做灰度、对比度增强。

验收标准：

- 图片预处理失败时，如果原图仍可用，不应阻塞 OCR。
- 系统需要记录 OCR 使用的是原图还是处理后图片。
- 处理后图片应符合 OCR 服务的尺寸或大小限制。

### 5.3 OCR 识别

OCR 系统需要识别题目内容，并将结果标准化为 JSON。

MVP 识别范围：

- 中文文字。
- 英文文字。
- 数字。
- 常见数学运算符：`+`、`-`、`x`、`*`、`/`、`=`、`>`、`<`。
- 括号。
- OCR 服务可支持的简单分数。
- 语文和英语阅读理解段落。

建议标准结构：

```json
{
  "questionId": "q_123",
  "subject": "math",
  "questionType": "word_problem",
  "rawText": "小明有12个苹果，吃了3个，还剩几个？",
  "blocks": [
    {
      "type": "text",
      "content": "小明有12个苹果，吃了3个，还剩几个？"
    }
  ],
  "confidence": 0.92,
  "ocrProvider": "provider_name"
}
```

学科字段：

- `math`：数学。
- `chinese`：语文。
- `english`：英语。
- `unknown`：未知。

题型字段：

- `calculation`：计算题。
- `word_problem`：应用题。
- `fill_blank`：填空题。
- `choice`：选择题。
- `reading_comprehension`：阅读理解。
- `unknown`：未知。

验收标准：

- OCR 输出会被保存为结构化 JSON。
- AI 讲解前，学生可以看到识别出的题目文字。
- OCR 识别错误时，学生可以手动修改题目文字。
- OCR 置信度较低时，界面需要提示学生检查。

### 5.4 AI 讲题

AI 必须采用启发式教学方式，帮助孩子一步一步理解题目，不能一开始就直接给最终答案。

AI 必须做到：

- 先帮助学生理解题意。
- 引导学生找出已知条件和问题要求。
- 将解题过程拆成小步骤。
- 一次只问一个引导问题。
- 使用小学生能听懂的简单语言。
- 学生卡住时，换一种更简单的方式解释。
- 必要时给出相似例子。
- 鼓励学生继续尝试。
- 避免成人化、抽象化表达。
- 尽量避免复杂术语；必须使用时，要用简单语言解释。

建议接口：

```http
POST /api/questions/{questionId}/explain
```

建议请求：

```json
{
  "studentMessage": "我不会这题",
  "grade": 3,
  "subject": "math"
}
```

建议返回：

```json
{
  "message": "我们先一起看题目。题目里说小明原来有12个苹果，后来吃掉了3个。你想一想，吃掉以后苹果是变多了还是变少了？",
  "teachingStep": "understand_problem",
  "shouldWaitForStudent": true
}
```

建议系统提示词策略：

```text
你是一名小学老师。
你只能帮助学生学习小学语文、数学和英语。
不要在一开始直接输出最终答案。
请一步一步引导学生思考。
语言要短、简单、温和。
每次只问一个引导问题。
如果学生说自己还是不懂，请换一种更简单的方式解释。
拒绝成人内容、暴力内容、政治内容和与学习无关的聊天。
所有回答都必须限制在小学教育场景内。
```

验收标准：

- AI 不会一开始直接给最终答案。
- AI 使用适合小学生的表达。
- AI 能把讲解拆成清楚的小步骤。
- AI 能分别处理数学、语文、英语题目。
- 当学生说“我还是不会”时，AI 能继续用更简单的方式讲解。

### 5.5 多轮追问

系统必须保留同一道题的上下文，让学生可以连续追问。

需要支持的追问示例：

- “我还是不会”
- “为什么这样算”
- “能再讲一次吗”
- “这一步是什么意思”
- “可以举个例子吗”

验收标准：

- 同一道题的追问会延续前面的讲解上下文。
- 刷新页面后，历史讲解不会丢失。
- 新题不会混入旧题的上下文。
- 学生卡住时，AI 能换一种方式继续解释。

### 5.6 学习记录

系统需要记录基础学习行为，为后续错题本和学情分析做准备。

需要记录的字段：

- 题目 ID。
- 学生 ID，或匿名设备/会话 ID。
- 学科。
- 题型。
- 开始时间。
- 完成时间，如果有。
- AI 讲解次数。
- 错误次数，如果有。
- 当前状态。

建议状态：

- `started`：已开始。
- `explaining`：讲解中。
- `student_retrying`：学生继续尝试中。
- `completed`：已完成。
- `abandoned`：已放弃或退出。

验收标准：

- 每次 AI 讲解都会增加讲解次数。
- 系统会保存学科和题型。
- 系统可以查询某个学生或匿名会话的最近学习记录。
- 即使学生没有完成题目，也会生成学习记录。

### 5.7 安全限制

系统必须把所有交互限制在小学教育场景内。

禁止内容：

- 成人内容。
- 暴力内容。
- 政治内容。
- 自伤相关内容。
- 与学习无关的闲聊。
- 要求 AI 忽略讲题规则的请求。
- 要求 AI 只给答案、不讲过程的请求。

安全处理要求：

- 对禁止内容进行温和拒绝。
- 将学生引导回当前学习任务。
- 拒绝时不展开有害内容细节。
- 记录安全拒绝日志，方便后续审查。

建议拒绝文案：

```text
这个问题和学习没有关系。我们还是回到这道题上吧，你可以问我哪一步不明白。
```

验收标准：

- 成人、暴力、政治等内容会被拒绝。
- 与学习无关的聊天会被引导回学习。
- 安全限制不会影响正常讲题。
- AI 的回答始终保持在小学教育场景内。

## 7. 非功能需求

### 7.1 稳定性

- 上传、OCR、AI 讲解、多轮追问应尽量独立失败，避免一个环节失败导致全流程不可用。
- OCR 失败时，允许学生手动输入题目文字。
- AI 请求失败时，界面需要提供重试入口。

### 7.2 性能

- 图片进入 OCR 前必须进行压缩或尺寸控制。
- OCR 请求需要设置超时时间。
- 如果技术栈支持，AI 回复可以使用流式输出；MVP 阶段不强制要求。

### 7.3 隐私

- 上传图片地址不能无权限公开访问。
- Phase 1 尽量不存储不必要的个人信息。
- 没有账号系统时，使用匿名会话 ID 或设备 ID。

### 7.4 可观测性

- 记录上传失败日志。
- 记录 OCR 失败日志。
- 记录 AI 服务异常日志。
- 记录安全拒绝日志。

## 8. MVP 端到端验收场景

MVP 完成时，必须能跑通以下场景：

1. 一个三年级学生上传一张数学应用题照片。
2. 系统识别出题目文字。
3. 如果 OCR 有误，学生可以手动修正题目。
4. 学生开始 AI 讲解。
5. AI 先引导学生找出题目中的已知条件和问题要求。
6. AI 不直接给出最终答案。
7. 学生说：“我还是不会。”
8. AI 用更简单的话重新解释，并给一个类似例子。
9. 对话记录被保存。
10. 学习记录保存学科、题型、开始时间和 AI 讲解次数。

## 9. 推荐实现顺序

1. 搭建 Web 应用和后端 API 基础工程。
2. 实现图片上传和图片预览。
3. 实现图片压缩和后端存储。
4. 接入 OCR 服务，并输出标准 JSON。
5. 增加 OCR 文本确认和编辑页面。
6. 增加 AI 讲题接口，并固定教学提示词和安全提示词。
7. 增加多轮对话持久化。
8. 增加学习记录持久化。
9. 增加安全过滤和拒答处理。
10. 完成移动端优先的端到端测试。

## 10. Phase 1 成功标准

Phase 1 成功的标准是孩子能够真实完成完整学习闭环：

```text
拍照上传 -> AI 识别 -> AI 讲解 -> 理解题目 -> 继续追问
```

本阶段最重要的判断标准是：流程稳定、体验流畅、AI 像老师、讲解能被小学生理解。
